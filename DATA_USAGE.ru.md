# Использование данных и воспроизводимость

Репозиторий содержит обезличенные результаты бенчмарков, производные таблицы, графики и материалы отчёта из личного GPU LLM runtime исследования.

## Что включено

- нормализованные таблицы результатов бенчмарков;
- избранные сводки метрик Prometheus/DCGM;
- сводки метрик LiteLLM и llama.cpp, безопасных для публикации;
- графики, построенные на основе артефактов бенчмарков;
- обезличенные описания методологии и заметки по запускам.

## Что не включено

- секреты, API-ключи, токены и учётные данные;
- приватные репозитории и внутренний инструментарий;
- данные и инфраструктура работодателя;
- клиентские данные;
- внутренние материалы компании;
- необезличенные логи, содержащие чувствительные пути, хосты или метки;
- сырые TSDB-снапшоты Prometheus (заархивированы в частном хранилище, не включены в публичный набор данных).

## Переиспользование

Данные предоставлены для обеспечения прозрачности и частичной воспроизводимости отчёта. При переиспользовании графиков, таблиц или выводов:

- сохраняйте указание авторства (см. [NOTICE.md](NOTICE.md));
- не представляйте exploratory-результаты как production-grade benchmark;
- учитывайте, что агентные результаты — однократные наблюдения, дисперсия пути не контролировалась.

## Граница публичной воспроизводимости

Публичного репозитория достаточно, чтобы проверить опубликованную методологию, нормализованные таблицы результатов, оговорки, аудит утверждений, графики и скрипты benchmark harness. Его также достаточно, чтобы увидеть, какие утверждения явно помечены как strong, moderate, weak, unsupported или ограниченные телеметрией.

Публичного репозитория недостаточно, чтобы независимо пересчитать каждое опубликованное число из сырых сэмплов. Полные сырые TSDB-снапшоты Prometheus, необезличенные логи, тела per-request запросов и часть run-local артефактов остаются в private archive, потому что могут содержать чувствительные метки, пути хоста, внутренние идентификаторы или операционные метаданные.

Публичные данные следует рассматривать как аудируемый пакет отчёта, а не как полный benchmark corpus. Утверждения, зависящие от предвычисленных сводок, private TSDB окон, атрибуции маршрутов dual-backend или per-scenario agent traces, требуют доступа к private archive или нового прогона с эквивалентной instrumentation.

## Что можно восстановить из private artifacts

Следующие пробелы потенциально можно закрыть, если проинспектировать и обезличить сырые артефакты из private archive:

- per-sample Prometheus/DCGM timeseries для Ada Q4 phase 1-3;
- per-row streaming TTFT для Ada 400k/2 и 524k/2, если они есть в run artifacts;
- range exports DCGM для dual RTX 5090, если TSDB содержит валидные аппаратные ряды;
- per-request route attribution для dual RTX 5090, если существуют structured routing logs или backend-specific request markers;
- per-scenario token counts, request counts и tool-call variance для агентных прогонов;
- более сильное доказательство dual-backend routing, только если в сырых данных есть route attribution или backend-specific logs.

## Предупреждение

Этот набор данных не является универсальным GPU benchmark корпусом. Он отражает конкретное железо, runtime-конфигурацию, файлы моделей и методологию тестирования, описанные в отчёте. Результаты ограничены протестированным стеком и не должны обобщаться за пределы условий, задокументированных в полном отчёте.
