# Исследование GPU Runtime: RTX 6000 Ada vs RTX 5090 (май 2026)

**[English version](README.en.md)**

---

## Что это такое

Данный репозиторий является исследовательским артефактом, документирующим практическое исследование поведения runtime-окружения при локальном LLM-инференсе с длинным контекстом на арендованных GPU Vast.ai. Исследование охватывает NVIDIA RTX 6000 Ada Generation (48 ГБ) и RTX 5090 (32 ГБ) — включая экспериментальную конфигурацию dual RTX 5090 — на моделях `Qwen3.6-27B-UD` в формате GGUF через стек `llama.cpp` + `LiteLLM` с телеметрией Prometheus/DCGM.

---

## Почему это важно

Практические вопросы, которые двигали исследованием:

- Насколько далеко можно расширить контекст на 48 ГБ Ada A6000 (300k / 400k / 524k) до упора в VRAM или проблемы со стабильностью?
- При одинаковом профиле `200k / 2`, насколько быстрее RTX 5090 по сравнению с Ada A6000?
- Помещается ли Q6-квантизация на Ada A6000 при 400k контексте и какова её операционная стоимость?
- Может ли LiteLLM маршрутизировать бенчмарковый трафик через два независимых llama.cpp-бэкенда на RTX 5090, и какая телеметрия отсутствует для этой топологии?
- Как поведение GPU отличается между синтетическими бенчмарками и реальными агентными нагрузками OpenCode?
- Какие метрики в стеке Prometheus/DCGM действительно информативны, какие являются прокси, а какие вводят в заблуждение?

## Ключевые результаты

Все утверждения ограничены протестированным стеком и run ID, задокументированными в полном отчёте. Классификацию силы доказательств см. в [аудите утверждений](data/claim-audit.md).

- **RTX 5090 — более быстрый воркер для профиля `200k / 2` Q4.** В синтетических строках с идентичным профилем RTX 5090 работал в 1.53×–1.63× быстрее Ada A6000 (wall-latency и decode TPS). Все пять агентных сценариев OpenCode также завершились быстрее, однако время агентных сценариев включает выполнение инструментов и не является чистой GPU-пропускной способностью.

- **Ada A6000 — лучший поддерживаемый эндпоинт для длинного контекста.** Успешно работал с Q4 при 300k/2, 400k/2 и 524288/2 — профилях, превышающих запас VRAM RTX 5090 (~4.6 ГБ при 200k/2). Ada Q4 VRAM масштабируется примерно на +4–5 ГиБ на каждые +100k контекста.

- **Ada Q4 400k/2 — лучший протестированный кандидат для агентных задач с длинным контекстом** в протестированном стеке: 36.2 ГиБ VRAM, ~12 ГиБ запаса, все пять агентных сценариев завершены. Сценарий D (heavy) сократился с 919 с при 200k/2 до 168 с при 400k/2, хотя точная причина не трассировалась.

- **Ada Q4 524288/2 — жизнеспособный, но агрессивный режим.** Все синтетические строки через 250k c2 завершены. Запас VRAM сокращается до ~7 ГиБ (DCGM-подтверждено). Агентные нагрузки при этом профиле не тестировались.

- **Q6 на Ada 400k/2 операционально жизнеспособен** для протестированного подмножества (+10–16% медленнее синтетически, +35–128% в однократных агентных сценариях, +7.1 ГиБ VRAM vs Q4). VRAM headroom сужается до ~5.2 ГиБ. Оценка качества не проводилась: нет доказательств улучшения качества выходных данных у Q6.

- **Dual RTX 5090 — топологический зонд, не бенчмарк.** LiteLLM обслуживал трафик через один эндпоинт на двух независимых llama.cpp-бэкендах, а приватные run logs показывают использование обеих GPU. Надёжность балансировки маршрутизации, per-request route attribution, данные DCGM и валидные метрики по отдельным бэкендам в этом запуске отсутствовали.

- **Агентные нагрузки ведут себя иначе, чем синтетические бенчмарки.** Утилизация GPU показывает бимодальный паттерн: близко к 100% во время инференса и близко к 0% во время выполнения инструментов. Wall-time сценариев включает задержку вызовов инструментов и нельзя конвертировать в TPS GPU.

- **Телеметрия Prometheus/DCGM является частью доказательной базы.** Данные RTX 5090 локально подтверждены (экспортированный JSON). Данные Ada Q6 вычислены из локального Prometheus. Фазы Ada Q4 1–3 используют предвычисленные сводки (полный TSDB в private archive).

## Роли кандидатов (наблюдения)

Рекомендации основаны только на протестированных профилях. Не является производственной политикой.

| роль | кандидат | примечания |
|---|---|---|
| Быстрый интерактивный воркер | RTX 5090, Q4 200k/2 | Лучшая протестированная wall-latency и decode TPS; 32 ГБ VRAM ограничивает масштабирование контекста |
| Эндпоинт с длинным контекстом | Ada A6000, Q4 400k/2 | Лучший протестированный кандидат; ~12 ГБ запаса; все агентные сценарии пройдены |
| Синтетика с огромным контекстом | Ada A6000, Q4 524288/2 | Проверено через 250k c2; ~7 ГБ запаса; агентное поведение не тестировалось |
| Эксперимент с более высокой точностью | Ada A6000, Q6 400k/2 | Операционально жизнеспособен; ~5 ГБ запаса; преимущество качества не доказано |
| Пул с маршрутизацией | 2× RTX 5090, LiteLLM | Экспериментально; быстро на прогретом кэше; надёжность маршрутизации не доказана |

## Что измерялось

| измерение | метрика | источник |
|---|---|---|
| End-to-end wall-latency | wall p50 / p95 по ячейке промпт/конкуренция | CSV хёрнесса |
| Streaming TTFT (на стороне клиента) | временная метка первого чанка в streaming HTTP | хёрнесс (начиная с фазы 2 и RTX 5090) |
| Время prefill на бэкенде | `prompt eval time` в логах llama.cpp | логи llama.cpp |
| TTFT на стороне сервера | гистограмма `litellm_llm_api_time_to_first_token_metric_bucket` | Prometheus (RTX 5090 + Ada Q6) |
| Decode-пропускная способность | predicted tokens per second | метрики llama.cpp + CSV хёрнесса |
| Занятость VRAM | `gpu_memory_used_mb` (DCGM NVML) | JSON Prometheus / предвычисленные сводки |
| Утилизация GPU | `gpu_utilization` SM % (DCGM) | JSON Prometheus |
| Потребляемая мощность | `gpu_power_draw` Вт (DCGM) | JSON Prometheus |
| Температура | `gpu_temperature` °C (DCGM) | JSON Prometheus |
| SM / memory clock | `gpu_sm_clock` / `gpu_memory_clock` МГц (DCGM) | JSON Prometheus |
| Утилизация слотов | `llama_busy_slots` / `llama_requests_deferred` | JSON Prometheus |
| Здоровье прокси | `litellm_requests`, `litellm_failed_requests`, `litellm_latency_p95` | JSON Prometheus |
| Wall-time агентного сценария | сквозное время сценария включая вызовы инструментов | выходные данные хёрнесса |
| Pass/fail агентного сценария | успешное завершение runtime + совпадение паттерна в логе | выходные данные хёрнесса |

## Ограничения

Данное исследование не устанавливает:

- **Качество выходных данных модели.** Оценка perplexity, MMLU, бенчмарки кода или задач не проводилась. Превосходство Q6 по качеству не доказано. Pass/fail агентных сценариев — это только runtime-завершение.
- **Универсальный рейтинг GPU.** Результаты ограничены протестированным стеком (llama.cpp + LiteLLM, конкретные GGUF-файлы, конкретные конфигурации хостов Vast.ai).
- **Поведение при производственной многопользовательской нагрузке.** Хёрнесс контролировал конкуренцию на уровне 1–6 параллельных запросов.
- **Воспроизводимые коэффициенты ускорения.** Агентные результаты — однократные наблюдения. Дисперсия пути не контролируется.
- **Модель стоимость/производительность.** Данные о цене аренды в этот отчёт не включены.
- **Подтверждение узкого места по пропускной способности VRAM.** Ускорение RTX 5090 согласуется с аппаратными различиями, но VRAM-to-SM bandwidth непосредственно не измерялась.
- **Надёжность маршрутизации dual RTX 5090.** Отсутствуют экспорты DCGM, per-backend метрики llama.cpp и LiteLLM TTFT. Это топологический зонд.

Сырые TSDB-снапшоты Prometheus хранятся в private archive (не публичны). Экспортированные санированные JSON-метрики для RTX 5090 доступны в `data/` (через `metrics_index.md`). Скрипты бенчмаркового хёрнесса — в `scripts/`. Бенчмарковые промпты используют псевдослучайный текст-заполнитель.

## Ресурсы и независимость исследования

Исследование выполнено независимо, в личное время, с использованием личных инструментов и арендованных либо находящихся под личным контролем вычислительных ресурсов.

В тестовых нагрузках и публикуемых артефактах не использовались инфраструктура работодателя, внутренние репозитории, конфиденциальные наборы данных, production-системы, клиентские данные или непубличные материалы компании.

Тестовые нагрузки были синтетическими, публичными, обезличенными или подготовленными специально для этого исследования.

---

## Ограничения исследования

Исследование ограничено доступными во время тестового окна аппаратными и runtime-профилями. Его следует рассматривать как практическое исследование поведения LLM runtime, а не как полный академический benchmark.

На объём исследования повлияли:

- ограниченное время аренды и тестирования;
- ограниченное число повторов для агентных сценариев;
- отсутствие production multi-user нагрузки;
- отсутствие полного перебора моделей и квантизаций;
- отсутствие формальной оценки качества ответов;
- наличие экспериментальных профилей, которые проверялись как exploratory, а не как финальные steady-state benchmark.

Эти ограничения зафиксированы в полном отчёте и должны учитываться при интерпретации выводов.
